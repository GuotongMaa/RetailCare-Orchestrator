## 0. 一句话架构主线(开场可用)

> RetailCare 是一个**状态可控、系统裁决**的电商售后 Agent。核心是三句话:
> **ReAct 做业务推进引擎**(模型自主决定下一步、可随对话切换业务),
> **LangGraph State 做可控机制**(记录已知/已验证/待办/已发生,防长链失忆),
> **系统而非模型做最终裁决**(身份、幂等、高危写都在代码层强制,模型的工具调用只是"提案")。
> 我做的升级,本质就是把这三条原则里"漏风"的地方逐个补严。

## 1. 身份越权(IDOR):从"模型传参"到"会话注入" 🔴

**场景**:每个工具(get_order / create_return_request …)都要 `user_id`,原设计是**在系统提示词里告诉模型当前用户、让模型把 user_id 填进每次工具调用**。

**难点 / 为什么危险**:真正流入所有权校验的 `user_id` 来自模型生成的 tool arguments。只要发生**提示注入**(订单备注、物流备注、RAG 策略块里藏指令)或模型**幻觉**出另一个 user_id,就能读/写**别人**的订单、发起退款——典型的水平越权(IDOR)。更隐蔽的是:guardrail 也救不了,因为它用的是同一个被污染的 user_id 去查 eligibility。这等于"系统裁决"原则在身份维度被**架空**了。

**方案**:把身份从"模型能表达的东西"里彻底拿掉。
- 工具 JSON schema 里**删除 `user_id`**(`registry.openai_spec` 用 `SERVER_INJECTED` 剥离)——模型连填的机会都没有。
- 在 `tools_node` 执行前,从**可信会话状态** `state["user_id"]` 注入,且**无会话用户则 fail-closed**(拒绝执行)。
- API 层 `user_id` 来自**已认证的 bearer token**,不再信任请求体。

**面试怎么讲(亮点)**:
- 体现"**信任边界**"思维:把不可信输入(模型/请求体)与可信主体(会话/token)分开,在边界处强制。
- "**fail-closed**" 设计:缺身份不是放行而是拒绝。
- 用一句话点出深度:"这个洞最隐蔽的地方在于 guardrail 也防不住,因为它复用了被污染的身份——所以修复必须在更底层的注入点。"

**现状**:✅ 已上线。认证已支持**生产 JWT 模式**(见 §9),MCP 已**按连接主体绑定身份**(见 §9)。

## 2. 系统派生幂等键 + int/float 重复发钱坑 🔴

**场景**:写操作要幂等键防重复。原设计让**模型生成**"一个稳定的 idempotency_key"。

**难点**:模型在重试/多轮里**不保证生成同一个 key** → `issue_compensation` 仅按 key 去重时会**重复发补偿**(真金白银)。

**方案**:幂等键改为**系统按业务身份确定性派生**(`derive_idempotency_key`):退货按 `(thread, user, order, item)`,补偿按 `(thread, user, reason, amount)`。模型无权设置。

**踩坑(独立 QA 抓到的)**:第一版直接哈希解析后的 `amount`,结果 `json.loads("15")` 是 int、`"15.0"` 是 float,`repr` 出 `"15"` vs `"15.0"` → **两个 key → 又重复发钱**。修复:金额归一化 `f"{float(x):.2f}"`。

**面试怎么讲(亮点)**:
- "**幂等性的关键是谁拥有 key**":交给模型(非确定性)是反模式,应由系统从业务不变量派生。
- 这个 int/float 坑特别能讲:"**幂等键对输入的规范化(normalization)和键本身一样重要**——类型/精度/大小写/空白任何不一致都会击穿幂等。"
- 顺带引出**取舍**:系统派生键会丢失"区分重试 vs 新动作"的能力,所以 `(reason, amount)` 相同的两笔合法补偿会被当成一笔(选择 dedup 优先于重复付款),需要不同 reason 区分。**会主动讲 tradeoff 是加分项。**

**现状**:✅ 已修 + 回归测试。

## 3. 事实源 / digest 分离 + 实时渲染(防 ReAct 失忆)⭐

**场景**:ReAct 链一长,模型会忘记"我之前做了什么、现在该干嘛",陷入迷路。想用 LangGraph state 当工作记忆把它"拉回来"。

**难点**:如果把状态摘要直接 append 进消息历史,**每轮都堆一份过期快照**,模型会锚定到错误的那一份——state 越丰富越容易记错。

**方案**:状态分两层,职责严格分离。
- **结构化字段**(intent/focus/findings…)= **唯一事实源**,checkpoint 持久化,**覆盖式**更新(证据类如 findings 用 merge reducer 累积)。
- **digest** = 事实源的**一次性投影**,在**调用 LLM 那一刻**实时渲染、拼进本次请求,但**绝不写回 messages**(用完即弃)。

**两个推论(面试金句)**:
1. "新状态覆盖旧的,内存里永远只有一份当前值;digest 因为从不入历史,**没有过期副本可被记错**。"
2. "**digest 只负责导航,不负责权威**——凡危险动作,系统都从 state/DB 重算,绝不采信模型的复述。digest 错了顶多是话术需澄清,**动不了执行**。"

**面试怎么讲(亮点)**:这是"**上下文工程**"的精彩案例——区分"给模型看的"和"系统据以决策的",并控制注入时机与生命周期。还能引出 LangGraph reducer 的取舍(单值覆盖 vs 证据合并)。

**现状**:✅ 已上线(≤300 token、focus-scoped、纯函数可单测)。

## 4. ReAct × LangGraph State 的分工哲学(架构主线)

**难点**:很多人要么纯 ReAct(灵活但失控、易迷路),要么纯固定工作流(可控但僵硬,不能随对话切业务)。售后场景**两者都要**。

**方案 / 取舍**:不为了加节点而加节点。LangGraph 主要用来提供**共享 state / checkpoint / interrupt / resume**,业务推进仍交给单个 ReAct agent。"是否值得加更多结构"由**评估数据**说话(项目有 ablation/回归 harness)。

**面试怎么讲(亮点)**:"我的设计哲学是 **restrained(克制)**——ReAct 给智能与灵活,State 给可控与记忆,系统给安全裁决;三者各司其职,而不是用流程图把 Agent 绑死。"能体现你有"何时该加结构、何时该克制"的判断力。

## 5. Guardrail 四态裁决 + 代码层重算(不信模型)

**场景**:高危写(退款/补偿)绝不能让模型说了算。

**方案**:写操作过 `guard_write`,**在代码里重新校验业务策略**,产出四态之一:`allow / confirm(HITL) / escalate(转人工) / block`。高危从不交给模型判断。

**面试怎么讲(亮点)**:"**防御纵深**——即使模型被注入或幻觉,真正的钱动作有一道独立于模型的、用 DB 真值重算的闸门。" 配合单一来源的高危名单(`registry.GATED_TOOLS`,避免硬编码名单与 `writes` 标志漂移)。

**现状**:✅ 已有;并把高危名单收敛为 registry 单一来源。

## 6. 独立 QA Agent:防"单 Agent 高估自己" ⭐

**场景**:实现完自测 72 个测试全绿,自我感觉良好。

**难点 / 洞察**:**单个 Agent(或开发者)会系统性高估自己的产出**,尤其安全/钱路径上的隐蔽 bug 自己测不出来。

**方案**:实现完后**派一个独立的对抗性 QA Agent**,明确"**不信任实现者的自评**",自己读代码、跑测试、写一次性探针去**证伪**各项声明。结果它抓到了第 2 条那个 int/float 重复发钱的 CRITICAL bug。

**面试怎么讲(亮点)**:这是**多 Agent 协作**与**工程质量文化**的真实案例——"用一个职责对立的 Agent 做红队,把'自测通过'和'真的安全'区分开"。非常有说服力,因为有真实抓到的 bug 佐证。

**现状**:✅ 已固化为工作流(后续实现完都会跑)。

## 7. HITL interrupt/resume 的 LangGraph 重跑陷阱(待做,D7)

**场景**:低值退款需用户确认(HITL),用 LangGraph `interrupt()` 暂停、`resume` 继续。

**难点**:`interrupt()` 在"遍历多个 tool_call"的循环内部,LangGraph 恢复时**从节点头部重跑**:① 该写之前的工具会被**重复执行**(靠读幂等 + 写去重兜底,但脆);② 一条消息含两笔待确认写时,确认结果可能**错配/丢失**;③ 确认值是个裸 "yes",**没和具体动作绑定**,期间 state 变了就可能批到别的动作。

**方案(计划)**:`interrupt` 时把 `pending_action.token`(动作指纹)写入 state;`resume` 时校验 token 一致**且** state 未变才执行;每节点最多 gate 一笔写,其余推迟下一轮。并补**多写 HITL 测试**(目前 0 覆盖)。

**面试怎么讲(亮点)**:能讲"**框架的恢复语义(re-run from node start)是把双刃剑**,幂等是兜底但不能当唯一防线;正确做法是把确认与具体动作做**因果绑定**,并把'同一轮只 gate 一笔写'作为结构性保证(消除按 interrupt 位置匹配 resume 的歧义)"。这是对框架理解深度的体现。

**方案(已实现)**:
- **单写闸**:`tools_node` 每轮最多 gate 一笔 confirm-write(`hitl_used` 标志),其余 confirm-write 返回 `deferred` 让模型下一轮重试 → 一轮内只有一个 `interrupt`,彻底消除位置错配。
- **token 绑定**:`interrupt` payload 带 `action_token`(动作指纹,re-run 可重算);`resume` 支持 `{"decision","token"}`,token 不匹配则 **fail-closed 视为拒绝**;裸 "yes"/"no" 保持兼容。
- 测试:多写只 gate 一笔(第二笔 deferred、不落库)、token 不匹配被拒、token 匹配执行。

**现状**:✅ 已实现并测试(C 阶段 C1)。`pending_action` 字段保留备用。

## 8. 其它待加固(C 阶段,简述)

- ✅ **退货窗口用真实时钟(C2,已做)**:原用冻结的 `seed.NOW`,上线即 bug。改为可注入 `clock.now()`,优先级:`set_now()` 覆盖(`seed()` 把"现在"钉在 seed 纪元保证测试确定性)→ `RETAILCARE_NOW` 环境变量 → 真实 `utcnow()`。讲点:"**确定性测试与生产正确性的冲突,用依赖注入/可注入时钟调和**——mock 世界的'现在'就是种子纪元,真实数据用真实时间。"
- ✅ **补偿累计上限(C3,已做)**:原只看单笔 ≥20 USD;新增**按用户滚动累计上限**(50 USD),
  且排除同一 idempotency_key 避免幂等重试自计。讲点:"**单点阈值挡不住小额累加**,风控要做累计维度;
  而累计求和要排除幂等重试,否则重试会被自己的历史顶过线。"
- ✅ **Trace 持久化(C4,已做)**:审计日志原为内存对象,重启即失。新增 `trace/store.py` 按
  `thread_id` 落盘(JSON 信封含 user_id),`/trace/thread/{id}` 重启后仍可查且按持久化的 user_id
  做归属校验。讲点:"**可审计性必须持久才算数**;且鉴权要绑在持久化记录上,而非易失的内存会话。"
- ✅ **并发硬化(C5,已做)**:checkpointer 单 sqlite 连接,WAL 已开但 `busy_timeout=0`,线程池下
  并发写会瞬间 `database is locked`。补 `PRAGMA busy_timeout=5000`(等待重试)。讲点:"**sqlite
  serialized 模式下共享连接是安全的,真正的坑是锁等待——WAL 让读写共存,busy_timeout 把'立即报错'
  变成'等一会儿'。**"
- ✅ **MCP 按连接绑定身份 + 认证 JWT 化(C6,已做)**:见 §9。

## 9. 认证演进(demo→JWT) + MCP 身份绑定(C6)

**场景**:第 1 条把身份从"模型/请求体"挪到了"会话边界",但 demo 用的是可伪造的 `Bearer demo-<user>`;而 MCP server 还把 `user_id` 当普通工具入参——是另一个越权面。

**难点**:① 真实认证要能验签、防 alg 混淆('none' 攻击)、查过期,且不引入重依赖;② MCP(stdio)没有天然的"已认证连接上下文",怎么把身份从工具参数里拿掉。

**方案**:
- **认证双模式**(`api/auth.py`):设了 `RETAILCARE_JWT_SECRET` → 走 **HS256 JWT 校验**(stdlib `hmac`,验签 + `exp` + **拒绝 alg≠HS256**,`sub` 即 user_id),demo 方案自动禁用;否则回退 demo 方案。`resolve_token` 是唯一切换点,换 RS256/JWKS/OIDC 不动 endpoint。
- **MCP 身份绑定**(`mcp_server/server.py`):工具签名**移除 `user_id`**,改由 `_mcp_user()` 从 `RETAILCARE_MCP_USER`(host 为每个已认证用户启一个 server 进程时注入)读取,缺失则 **fail-closed**。客户端/模型再也传不进别人的 user_id。

**面试怎么讲(亮点)**:
- "**认证逻辑收敛到单一可替换点**(`resolve_token`),demo 与生产同构、只换实现。"
- "**alg 混淆('none' token)是 JWT 经典坑**,我显式校验 alg 并用常数时间比较签名。"
- "MCP 的身份绑定和 in-process agent **同一个哲学**:身份由服务端绑定,绝不是调用方/模型的入参——把 D2 贯彻到每一个入口。"

**现状**:✅ 已实现并测试(JWT 有效/篡改/过期/none、demo 回退、MCP 绑定/fail-closed)。MCP 写操作的
幂等键也已**系统派生**(从签名移除 `idempotency_key`,由 `(session, user, 业务键)` 派生,与 agent 路径统一)。
残余:RS256/JWKS。

## 附:可复用的"金句/原则"清单(速记)

- 把**不可信输入**与**可信主体**在边界处分开,缺身份就 **fail-closed**。
- **幂等键由系统按业务不变量派生**,且**规范化输入**(类型/精度/大小写/空白)。
- **给模型看的 ≠ 系统据以决策的**;digest 是导航,不是权威。
- 危险动作一律**用 DB 真值重算**,不采信模型复述。
- **克制地加结构**:ReAct 给灵活,State 给可控,系统给安全。
- **独立对抗 QA** 把"自测通过"和"真的安全"区分开。
- 框架的**恢复语义**要吃透;幂等是兜底,不是唯一防线。
