日期：2026-06-21

## 本章核心理解

不要把 Agent 简单理解成“接了大模型 API 的聊天机器人”。

更准确的理解是：

```text
AI Agent = 大模型认知核心 + 状态感知 + 任务规划 + 记忆 + 工具 + 执行 + 反馈/反思
```

它的重点不是“会说话”，而是能在一个受约束的环境里完成多步任务。

## 六大组成与 RetailCare 对应

| 组成 | 中英对照 | 概念解释 | RetailCare 中的体现 |
|---|---|---|---|
| 感知 | 感知 / Perception | 把用户输入、接口返回、工具结果、页面或文件等信息转成模型可用的任务状态。 | 当前项目主要感知文本输入、系统提示、订单/物流/政策等工具返回结果，还没有多模态图片或页面结构感知。 |
| 规划 | 规划 / Planning | 把目标拆成步骤，决定下一步做什么；可以一次性计划，也可以每步根据新观察重新规划。 | RetailCare 是轻量规划：模型不会先输出完整长计划，而是在每一轮决定下一步调用哪个工具、是否澄清、是否升级人工。 |
| 记忆 | 记忆 / Memory | 短期记忆保存当前任务上下文和工具轨迹；长期记忆保存用户画像、文档知识、向量库或图数据库。 | 这个项目以短期记忆为主：会话消息、工具轨迹、checkpoint、trace、ticket summary。长期记忆较弱，没有完整用户画像或跨任务偏好记忆；政策知识/RAG 更接近外部知识库，不等同于用户长期记忆。 |
| 工具 | 工具 / Tools | 对外部世界可执行或可查询的操作抽象，需要清晰名称、描述、参数 Schema 和返回结构。 | 项目有 8 个售后工具：查订单、查物流、查政策、查优惠券、判断退货资格、创建退货、发补偿、升级人工。 |
| 执行 | 执行 / Action | 真正调用工具或改变环境，并处理超时、重试、幂等、错误恢复等工程问题。 | 项目通过工具层执行真实业务动作；写操作有 guardrails、人类确认、幂等键和故障恢复，避免重复退款或越权操作。 |
| 反思 | 反思 / Reflection | 对失败或质量不佳的结果做自省、纠错、换策略，常和评估器、检查清单、重试机制配合。 | 项目没有完整 Reflexion 框架，但有工程化的“弱反思”：工具错误会回写给模型，模型可重试或换策略；eval、trace、error taxonomy 用于离线复盘。 |

## RetailCare 到底是什么类型的 Agent

RetailCare 更准确地说是：

```text
受约束的任务型售后 Agent
```

它不是开放世界的 AutoGPT 式 Agent，也不是多模态通用助手。它服务一个明确业务域：电商售后/客服，尤其是订单查询、物流、退货退款、补偿、投诉升级。

所以它的优势不是“什么都能做”，而是：

- 工具边界清楚；
- 业务规则明确；
- 高风险动作有护栏；
- 过程可追踪；
- 任务能评测；
- 出错能恢复。

|---|---:|---|
| Agent 不是单次回复，而是会拆任务、查资料、调用工具、汇总结果的系统 | 第 10-11 页 | 可以用退货退款流程举例：用户说要退货，系统不会直接生成安慰话术，而是查订单、判断退货资格、查政策、必要时创建退货单或升级人工。 |
| `Agent ≈ LLM + Planning + Memory + Tools` | 第 10-11 页 | LLM 负责理解和决策；Planning 体现在每轮选择下一步工具或澄清；Memory 体现在会话状态、checkpoint、trace、summary；Tools 体现在订单、物流、政策、退货、补偿、人工升级工具。 |
| 普通 LLM 调用是开环生成，Agent 是闭环决策 | 第 11 页 | RetailCare 的工具结果会回到上下文，模型根据 observation 继续判断。比如 eligibility 返回高价值退款需要人工审核，下一步就不应继续创建退货，而应升级人工。 |
| Agent 的自主不是失控，而是在约束空间内决策 | 第 12 页 | 项目有工具白名单、参数 schema、guardrails、HITL、人类确认、最大步数、trace 审计。模型能选工具，但不能绕过写操作安全规则。 |
| ChatBot、LLM Chain、Agent 的区别 | 第 13-14 页 | RetailCare 如果只回答“请联系客服”就是 ChatBot；如果固定执行“查订单 -> 生成答复”就是 Chain；现在它能根据用户输入和工具结果动态选择查政策、判断资格、创建退货或升级人工，因此是 Agent。 |
| RAG + Chat 不一定是 Agent | 第 14 页 | RetailCare 的 policy RAG 如果只是“检索一次政策再回答”，只能算增强型 Chat；当它和退货资格判断、工具执行、guardrail 决策、人工升级形成闭环时，才体现 Agent 特征。 |
| 感知 / Perception | 第 15 页 | 项目当前感知文本输入、系统提示、工具返回、业务规则和政策片段；没有图片、语音、页面 DOM 等多模态感知，所以不能夸成多模态 Agent。 |
| 规划 / Planning | 第 15-16 页 | 项目不是完整 Planner-Executor，而是 ReAct 式轻量规划：每一轮根据当前状态决定下一步动作。退货场景中常见计划是先确认订单和商品，再判断资格，再决定写操作或升级。 |
| 记忆 / Memory | 第 15-16 页 | 项目以短期记忆为主：当前消息、工具轨迹、checkpoint、trace、ticket summary。长期用户画像和偏好记忆不是当前重点。政策 RAG 是长期知识来源，但不是用户长期记忆。 |
| 工具 / Tools | 第 15-16 页 | 项目有 8 个带 Pydantic schema 的工具，覆盖订单、物流、政策、优惠券、退货资格、创建退货、补偿、人工升级。面试时可以强调工具名称、描述、参数和返回结构都是显式契约。 |
| 执行 / Action | 第 15 页 | 项目不是只“建议你可以退款”，而是真的能触发 create_return_request、issue_compensation、escalate_to_human 等动作；写操作有幂等键、确认、恢复和审计。 |
| 反思 / Reflection | 第 15-16 页 | 项目没有完整 Reflexion 子系统，不能夸大。但它有工具错误回写、recovery、trace、eval、error taxonomy，用工程化复盘替代一部分在线反思。 |
| Agent 工作流程：听懂 -> 拆活 -> 动手 -> 对账 -> 交卷 | 第 17 页 | 可以用 hero refund flow 讲：听懂退货诉求；拆成查订单/查政策/判断资格；调用工具；根据结果处理冲突或人工确认；最后给用户票据或升级说明。 |
| 停止条件和最大步数 | 第 17、22 页 | 项目有 `MAX_STEPS`，模型无工具调用时结束，避免无限循环。这可以作为“生产 Agent 必须有限步”的例子。 |
| Agent 分类：反应式、基于模型、目标型、效用型、学习型 | 第 18-19 页 | RetailCare 是混合型：主要是目标型任务 Agent，目标是完成售后任务；部分规则是反应式；评估报告和 Pareto 报告体现了效用型权衡，例如质量、成本、延迟。它不是在线学习型 Agent。 |
| 应用场景：智能客服最适合高频、可工具化、可评估场景 | 第 19-20 页 | RetailCare 正是智能客服/售后场景，任务高频、业务规则清楚、工具可执行、结果可评测，非常适合做 Agent 项目案例。 |
| 客服场景降低幻觉：RAG 溯源、禁止无引用断言、未知转人工 | 第 20 页 | 项目要求不要编造订单、退款金额或政策；政策有版本化 chunk；不确定、高价值、争议场景升级人工。 |
| Agent 挑战：幻觉、安全、成本、可解释、评估困难 | 第 20-21 页 | RetailCare 的应对分别是：工具/RAG grounding；guardrails/HITL/最小权限；usage/cost 统计和 Pareto；trace 日志；refund benchmark、pass^k、compliance metrics、error taxonomy。 |

## ChatBot、LLM Chain、Agent 的区别

资料第 13-14 页的核心差异可以这样理解：

| 类型 | 控制方式 | RetailCare 对比 |
|---|---|---|
| 聊天机器人 / ChatBot | 主要根据上下文生成回复。 | 如果 RetailCare 只回答“你可以联系客服”，那只是 ChatBot。 |
| 大模型链 / LLM Chain | 开发者预先写好固定步骤，模型按流程填内容。 | 如果流程永远是“查订单 -> 查政策 -> 生成答复”，且没有动态选择，就更像 Chain。 |
| 智能体 / Agent | 模型根据当前观察选择下一步动作，可以查工具、等待结果、重试、澄清或结束。 | RetailCare 会根据用户问题动态选择工具，并根据工具结果继续决策，所以具备 Agent 特征。 |

## 记忆系统的判断

用户问题：“我的项目应该是短期记忆为主吧？”

答案：是的，当前 RetailCare 主要是短期记忆和任务级记忆。

具体来说：

- 短期记忆 / Short-term Memory：当前会话消息、系统提示、工具返回结果、LangGraph state。
- 工作记忆 / Working Memory：当前退货任务中已经确认的 order_id、item_id、reason、eligibility、refund_amount。
- 轨迹记忆 / Trace Memory：trace 记录工具调用、结果、错误、guardrail 决策和 HITL 中断。
- 摘要记忆 / Summary Memory：`TicketSummary` 从 trace 中提取工单摘要，帮助低成本理解当前任务状态。
- 长期记忆 / Long-term Memory：目前不强。项目没有持续学习用户偏好、跨会话画像、长期个性化偏好库。政策 RAG 是外部知识库，算长期知识来源，但不是“用户长期记忆”。

这个判断很重要，因为面试时不能夸大：

```text
我们实现了以短期会话状态、checkpoint 和 trace 摘要为主的任务记忆；
长期用户画像和偏好记忆目前不是主线，只保留为后续扩展方向。
```

## 调用链路

```text
用户输入
-> 感知：解析用户意图和关键信息
-> 规划：决定下一步查订单、查政策、判断资格、澄清或升级
-> 工具：调用订单/物流/政策/退货等工具
-> 执行：工具真正查询或写入业务状态
-> 观察：工具结果回到模型上下文
-> 记忆：消息、工具轨迹、checkpoint、trace、summary 保存任务状态
-> 反思/纠错：工具错误、guardrail 拦截或 eval 复盘推动下一步调整
-> 最终答复或人工升级
```

## 面试表达

RetailCare 是一个电商售后场景的任务型 AI Agent。资料里说 Agent 的核心不是单次文本生成，而是感知、规划、记忆、工具、执行和反馈组成的闭环。对应到我的项目，用户输入和工具返回构成感知；模型在每一轮决定下一步查订单、查政策、判断退货资格还是升级人工，这属于轻量规划；项目以短期记忆为主，用会话状态、checkpoint、trace 和 ticket summary 保存当前任务轨迹；工具层封装了 8 个售后操作，并通过 schema、guardrails、HITL 和幂等机制保证执行安全。反思方面没有做完整 Reflexion 框架，但通过工具错误回写、trace、eval 和错误分类做工程化复盘。因此它不是普通 ChatBot，也不是固定 Chain，而是一个受约束、可追踪、可评测、偏生产实践的售后 Agent。
